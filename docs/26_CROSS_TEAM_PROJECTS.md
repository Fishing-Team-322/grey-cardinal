# Межкомандные проекты Grey Cardinal

Статус документа: production specification  
Версия: 1.0  
Владелец домена: Grey Cardinal / Project Collaboration

## 1. Цель

Добавить в Grey Cardinal слой межкомандных проектов, который позволяет нескольким
командам одной компании совместно планировать и выполнять работу, не переводя
весь продукт на проектную модель.

Команды, которым проекты не нужны, продолжают работать с обычными командными
задачами, Telegram-чатом и одной доской без дополнительных обязательных шагов.

## 2. Принципы продукта

1. Проект является необязательным контейнером над задачами.
2. Каждая задача сохраняет одну ответственную команду.
3. У задачи может быть несколько участвующих команд и исполнителей.
4. Проект существует только внутри одной компании.
5. Grey Cardinal является источником истины. YouGile является синхронизируемым
   внешним представлением.
6. Создание AI-проекта всегда проходит через сохраняемый предпросмотр.
7. Проектные чаты и Telegram topics необязательны.
8. Обычный командный чат продолжает работать без выбора проекта.
9. При неоднозначном контексте бот не создаёт задачу молча, а запрашивает выбор.
10. Все внешние операции идемпотентны и не должны приводить к потере локальных данных.

## 3. Роли и доступ

### Директор

- видит и управляет всеми проектами компании;
- создаёт проекты с любым набором команд;
- выбирает ведущую команду;
- запускает и повторяет синхронизацию YouGile;
- управляет проектными чатами;
- видит бюджет, загрузку и межкомандные риски.

### Руководитель

- видит планировщик и проекты своей компании;
- создаёт проект, если ведущей является хотя бы одна управляемая им команда;
- редактирует проект, если управляет ведущей или участвующей командой;
- назначает только участников проекта;
- видит бюджет и прогнозы проекта.

### Сотрудник

- видит проекты, в которых участвует лично или через команду;
- видит проектные задачи, доступные его командам;
- меняет статус назначенных ему задач;
- не видит черновики, бюджетные сценарии и внутренние рекомендации планировщика.

## 4. Жизненный цикл

### AI-предпросмотр

1. Руководитель или директор вводит описание результата.
2. Система декомпозирует работу на задачи.
3. Система оценивает трудозатраты, бюджет, P50/P90 и влияние на загрузку.
4. Система предлагает название, ведущую команду, участвующие команды,
   исполнителей и сроки.
5. Результат сохраняется как versioned draft с ограниченным сроком жизни.
6. Пользователь редактирует название, сроки, состав команд и задачи.
7. Сервер повторно валидирует принадлежность всех сущностей одной компании.

### Создание

1. Draft преобразуется в `company_project`.
2. Создаются проектные команды и участники.
3. Создаются задачи, связи команд и исполнителей.
4. Пишется audit log.
5. Локальная транзакция фиксируется до обращения к внешним системам.
6. При включённой синхронизации создаётся проект YouGile.
7. Ошибка YouGile не откатывает локальный проект и отображается как sync warning.

### Статусы проекта

- `draft` — внутренний черновик проекта;
- `active` — проект выполняется;
- `paused` — работа временно приостановлена;
- `completed` — проект завершён;
- `cancelled` — проект отменён.

## 5. Модель данных

### `company_projects`

- компания;
- код и название;
- описание и ожидаемый результат;
- владелец и создатель;
- статус;
- даты начала и завершения;
- бюджетный диапазон;
- источник создания;
- состояние внешней синхронизации.

### `company_project_drafts`

- исходное описание;
- сгенерированный blueprint;
- версия;
- создатель;
- срок жизни;
- source team для выбора LLM-конфигурации.

### `project_teams`

- проект;
- команда;
- роль: `lead`, `contributor`, `observer`;
- процент плановой загрузки;
- статус участия.

### `project_members`

- проект;
- пользователь;
- команда происхождения;
- проектная роль;
- процент плановой загрузки;
- активность участия.

### Совместные задачи

`tasks.team_id` остаётся ответственной командой.  
`tasks.company_project_id` связывает задачу с новым проектом.

Дополнительные таблицы:

- `task_teams` — участвующие команды и их роль;
- `task_assignees` — несколько исполнителей и их роль.

Существующий `tasks.assignee_id` остаётся главным исполнителем для обратной
совместимости со старым UI, Telegram-командами и YouGile adapter.

### Интеграции

- `project_external_links` — связь проекта с YouGile;
- `project_chat_bindings` — отдельный чат или topic проекта;
- `collaboration_events` — события межкомандной работы для карты и достижений.

## 6. API

### Планировщик

- `POST /api/companies/{company_id}/project-drafts/preview`
- `GET /api/project-drafts/{draft_id}`
- `PATCH /api/project-drafts/{draft_id}`
- `POST /api/project-drafts/{draft_id}/create`

### Проекты

- `GET /api/companies/{company_id}/projects`
- `GET /api/teams/{team_id}/projects`
- `GET /api/projects/{project_id}`
- `PATCH /api/projects/{project_id}`
- `POST /api/projects/{project_id}/yougile/sync`
- `POST /api/projects/{project_id}/chat-bindings`
- `DELETE /api/projects/{project_id}/chat-bindings/{binding_id}`

### Задачи

Payload задачи проекта содержит:

- `owner_team_id`;
- `team_ids`;
- `assignee_ids`;
- `estimated_hours`;
- `deadline`;
- `dependencies`.

Сервер отклоняет исполнителя, который не входит в проект.

## 7. YouGile

### Маппинг

```text
Grey Cardinal company project -> YouGile project
Grey Cardinal project workflow -> YouGile board
Grey Cardinal task -> YouGile task
Grey Cardinal project member -> YouGile project user
```

Для проекта выбирается интеграция ведущей команды. Все команды должны относиться
к одной компании Grey Cardinal. Пользователи сопоставляются с YouGile по email.

### Создание

1. Создать YouGile project с пользователями и ролями.
2. Создать board.
3. Создать колонки `К выполнению`, `В работе`, `На проверке`, `Готово`.
4. Создать задачи с несколькими исполнителями, deadline и плановым временем.
5. Сохранить external IDs.

### Ошибки

- локальный проект не удаляется;
- link получает `error`;
- сохраняется безопасное описание ошибки;
- UI предлагает повторить синхронизацию;
- повторный запуск использует существующий link и не создаёт второй проект.

## 8. Telegram

Контекст определяется в порядке:

1. явная project chat/topic binding;
2. ID существующей задачи;
3. явное название или код проекта;
4. контекст последних сообщений;
5. единственный активный проект команды;
6. запрос выбора пользователю.

В проектном контексте список допустимых исполнителей ограничивается
`project_members`. Командный чат без project binding работает по старым правилам.

## 9. Grey Board

Grey Board получает переключатель:

- `Команда` — все задачи команды, включая задачи без проекта;
- `Проекты` — список проектов команды;
- выбранный проект — единая доска задач нескольких команд.

Карточка проектной задачи показывает:

- проект;
- ответственную команду;
- участвующие команды;
- несколько исполнителей;
- зависимости;
- состояние YouGile sync.

## 10. Карта команд

Карта имеет режимы:

- `Структура`;
- `Проекты`;
- `Взаимодействие`.

Узлы:

- команды;
- активные проекты.

Рёбра строятся по `project_teams`, `task_teams` и `collaboration_events`.
Вес ребра учитывает количество совместных задач, закрытия и блокирующие передачи.

## 11. Достижения

События:

- `cross_team_task_completed`;
- `project_completed`;
- `handoff_completed`;
- `project_helped`;
- `cross_team_blocker_resolved`.

Достижения:

- «Мост между отделами»;
- «Общий результат»;
- «Надёжный партнёр»;
- «Быстрая передача»;
- «Синхронная работа».

Награды начисляются только по идемпотентным событиям с уникальным ключом.

## 12. Onboarding

Создатель компании выбирает рабочую модель:

- только команда;
- команды и проекты.

После создания команды доступны необязательные шаги:

- добавить стартовые задачи;
- пригласить сотрудников;
- подключить Telegram;
- подключить YouGile;
- создать первый AI-проект.

Пропуск проектного шага не ограничивает продукт.

## 13. Нефункциональные требования

- все tenant-запросы проверяют company/team/project scope;
- draft payload повторно валидируется при создании;
- внешние credentials не попадают в API, audit log и sync errors;
- создание проекта локально атомарно;
- внешняя синхронизация retry-safe;
- изменения покрыты unit/API tests;
- миграция совместима с PostgreSQL и SQLite test metadata;
- списки проектов имеют ограничение и предсказуемую сортировку;
- проектные endpoints не ухудшают старые командные сценарии.

## 14. Метрики и наблюдаемость

- число созданных и активных проектов;
- preview-to-create conversion;
- доля межкомандных задач;
- время закрытия межкомандной зависимости;
- количество sync errors;
- количество ambiguous Telegram context requests;
- число завершённых совместных проектов.

## 15. Критерии приёмки

1. Обычная задача без проекта создаётся и синхронизируется как раньше.
2. Директор создаёт AI-preview и проект с двумя командами.
3. Проект содержит задачи разных ответственных команд.
4. Одна задача может иметь несколько исполнителей.
5. Сотрудник вне проекта не может быть назначен на его задачу.
6. Проект виден участникам и скрыт от посторонних компаний.
7. YouGile sync создаёт project, board, columns и tasks.
8. Ошибка YouGile не удаляет локальные данные.
9. Карта компании показывает проектные связи команд.
10. Onboarding позволяет как включить, так и полностью пропустить проекты.
