# Дизайн: Эмоциональный портрет сотрудников

> Статус: **дизайн** (Bucket B). Кода пока нет.
> Решение пользователя: использовать **все источники** (текст, аудио, видео,
> поведение). Легитимность — через **по-отдельный opt-in отдела**: директор/
> руководитель включает функцию для отдела и тем самым принимает условия
> записи/анализа; участников уведомляем.

## Зачем

Отслеживать эмоциональное состояние сотрудников из разных сигналов, чтобы
руководитель видел выгорание/перегрузку/демотивацию **на агрегированном уровне**
и мог вовремя реагировать. Это же — топливо для тамагочи-геймификации (см.
`gamification-tamagotchi.md`).

## Согласие и юр. рамка (152-ФЗ / GDPR)

- Анализ **выключен по умолчанию**. Включается **на уровне отдела/команды**:
  флаг `team.board_config.emotion_analysis = { enabled, enabled_by, enabled_at,
  sources: {text, audio, video, behavior} }`.
- Включить может только **director** компании или **manager** команды
  (`_actor_can_manage`). Факт включения = согласие ответственного лица на
  обработку и принятие условий; в `audit_logs` пишем кто/когда/какие источники.
- Участникам отдела бот отправляет уведомление: какие источники анализируются,
  как отключить персонально (право на отказ от video/audio при сохранении
  поведенческих метрик).
- **Сырьё не храним**: кадры видео и аудио-семплы обрабатываются «на лету» →
  сохраняем только производные числовые метрики. Ретеншн метрик — настраиваемый
  (по умолчанию 90 дней), потом агрегируется/удаляется.
- Менеджеру доступны **только агрегаты** (по отделу и тренды по сотруднику), не
  «сырые эмоции по секундам».

## Источники сигналов

| Источник | Откуда берём | Что извлекаем | Готовность |
|---|---|---|---|
| **Текст чата** | `chat_messages` + LLM | valence (позитив/негатив), стресс-маркеры, тон | LLM уже есть → дёшево |
| **Поведение** | `tasks`, синки, дедлайны | просрочки, пропуски вечернего синка, скорость, активность | данные уже в БД |
| **Аудио созвонов** | audio-worker / ASR pipeline | просодия: темп, паузы, тон голоса | нужен новый анализатор поверх ASR |
| **Видео с камеры** | meeting/telemost recorder | facial affect (агрегат по выражению лица) | новый, тяжёлый источник |

Поэтапно: **текст → поведение → аудио → видео** (по возрастанию сложности и
чувствительности).

## Модель данных

```python
class EmotionSignalModel:            # «сырой» сигнал из одного источника
    id, team_id, user_id, source     # 'chat_text' | 'call_audio' | 'call_video' | 'behavior'
    valence  Float                   # -1..1 (негатив..позитив)
    arousal  Float                   # 0..1 (спокойствие..возбуждение)
    stress   Float                   # 0..1
    confidence Float
    ts, source_ref(JSON)             # ссылка на meeting/сообщение (без сырья)

class EmotionPortraitModel:          # агрегат (rollup) на сотрудника/период
    id, team_id, user_id, period_start, period_end
    valence_avg, arousal_avg, stress_avg, trend  # up | flat | down
    sample_count, sources(JSON)      # вклад каждого источника
```

Агрегат на отдел считается из `EmotionPortrait` по членам команды (без раскрытия
индивидуальных «сырых» значений в командных вью).

## Пайплайн

```
opt-in отдела включён?
  ├─ text:     ingest chat_message → LLM sentiment → EmotionSignal(chat_text)
  ├─ behavior: ночной job по tasks/синкам → EmotionSignal(behavior)
  ├─ audio:    ASR-транскрипт + просодия → EmotionSignal(call_audio)
  └─ video:    recorder facial-affect (агрегат/сек) → EmotionSignal(call_video)
        ↓ периодический rollup
   EmotionPortrait (сотрудник) → агрегат отдела → тамагочи + менеджерские вью
```

Текстовый сигнал встраивается в **существующий** semantic-пайплайн: добавить в
`SemanticParseResult` опциональный блок `affect {valence, stress}` и при
включённом флаге писать `EmotionSignal`. Это переиспользует уже работающий
LLM-вызов (без отдельного прохода).

## Приватность по дизайну (сводка)

- Per-department opt-in + персональный opt-out на video/audio.
- Только производные метрики, сырьё не хранится.
- Менеджеру — агрегаты и тренды, не посекундная «расшифровка эмоций».
- Полный аудит включения/выключения и доступа.

## Этапы реализации

1. Флаг отдела + согласие/уведомления + аудит (без анализа).
2. Текстовый сентимент в semantic-пайплайне → `EmotionSignal` + rollup.
3. Поведенческие сигналы (ночной job на готовых данных).
4. Аудио-просодия поверх ASR.
5. Видео facial-affect в recorder (самый чувствительный — последним).
6. Менеджерские агрегаты + связь с тамагочи.
